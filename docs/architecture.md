# Архитектура проекта

## Цель архитектуры

Сделать систему, в которой 4 участника одновременно работают над разными частями без блокировки друг друга и с прозрачными контрактами между модулями.

## Выбранный подход

- Тип архитектуры: модульный монолит (backend) + отдельный frontend.
- Формат репозитория: монорепо.
- Контракты: HTTP API + общий OpenAPI-файл в `packages/contracts/openapi.yaml`.
- Принцип: каждый модуль имеет собственную папку, интерфейсы и тесты.

## Технологический стек v1

- Frontend: Next.js + TypeScript
- Backend API: FastAPI + Pydantic
- База данных (следующий шаг): PostgreSQL
- Интеграции: provider-based подход (Google, GitHub, LMS)
- Тесты: pytest (backend), ESLint (frontend)

## Слои backend

1. API слой (`app/api`)
- Роуты и валидация входящих запросов.
- Не содержит сложной бизнес-логики.

2. Модули предметной области (`app/modules`)
- `intelligence`: планировщик, scoring, prime-task.
- `knowledge`: модели задач/событий и репозиторий.
- `integrations`: провайдеры внешних сервисов и sync-операции.

3. Composition root (`app/container.py`)
- Сборка зависимостей и инициализация модулей.

## Поток данных

1. Пользователь создает/обновляет задачу через frontend.
2. Frontend вызывает API `tasks`.
3. Модуль knowledge сохраняет изменения и помечает расписание как dirty.
4. API вызывает `schedule/rebuild`.
5. Модуль intelligence формирует план и prime-task.
6. Frontend получает `schedule/today` и отображает рекомендации.
7. Интеграции синхронизируют внешние источники и тоже помечают расписание dirty.

## Границы ответственности (важно для параллельной разработки)

- `apps/web` не меняет бизнес-алгоритмы планирования.
- `intelligence` не работает с SQL напрямую, только с абстракцией репозитория.
- `knowledge` отвечает за схемы данных, хранение и миграции.
- `integrations` не меняет UI, только API и внутренние adapters.

## Почему не микросервисы

Для курсового важнее скорость разработки и стабильная интеграция. Модульный монолит проще запускать и дебажить, при этом границы модулей уже подготовлены для возможного выделения в отдельные сервисы позже.

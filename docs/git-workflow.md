# Git workflow

## Ветки

- `main` — стабильная версия, только через PR из `develop`.
- `develop` — интеграционная ветка команды.
- feature-ветки каждого участника:
  - `feat/intelligence/*`
  - `feat/frontend/*`
  - `feat/knowledge/*`
  - `feat/integrations/*`

## Процесс

1. Создать feature-ветку от `develop`.
2. Сделать небольшой законченный кусок (1 PR = 1 цель).
3. Прогнать локальные проверки.
4. Открыть PR в `develop`.
5. После ревью выполнить squash merge.

## Правила PR

- Максимум 300-500 строк измененного кода в одном PR (если возможно).
- Если меняется контракт API, в PR обязательно:
  - обновление `docs/api-contract.md`
  - обновление `packages/contracts/openapi.yaml`
- В описании PR указывать:
  - что сделано
  - какие риски
  - как проверить

## Конфликты и merge

- Если конфликт в чужом модуле, не решать в одиночку.
- Созвон 10-15 минут, фиксируете единое решение, после чего один человек пушит итог.
- В `main` мержите только после smoke-теста всей команды.
